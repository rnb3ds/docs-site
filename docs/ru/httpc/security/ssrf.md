# Защита от SSRF

SSRF (Server-Side Request Forgery, подделка серверных запросов) — это атака, при которой злоумышленник использует сервер для отправки запросов во внутреннюю сеть. HTTPC включает защиту от SSRF по умолчанию.

## Поведение по умолчанию

```go
cfg := httpc.DefaultConfig()
// AllowPrivateIPs = false → по умолчанию блокируются все приватные IP
```

Блокируемые диапазоны IP по умолчанию:

| Диапазон | CIDR | Описание |
|----------|------|----------|
| IPv4 loopback | `127.0.0.0/8` | localhost |
| Класс A приватные | `10.0.0.0/8` | Внутренняя сеть |
| Класс B приватные | `172.16.0.0/12` | Внутренняя сеть |
| Класс C приватные | `192.168.0.0/16` | Внутренняя сеть |
| Link-local | `169.254.0.0/16` | Автоконфигурация |
| IPv6 loopback | `::1/128` | localhost |
| IPv6 локальные | `fc00::/7` | Уникальные локальные адреса |
| IPv6 link-local | `fe80::/10` | Link-local |

## Исключения CIDR

В микросервисной среде может потребоваться доступ к внутренним сервисам:

```go
cfg := httpc.DefaultConfig()
cfg.Security.SSRFExemptCIDRs = []string{
    "10.0.0.0/8",       // Внутренний VPC
    "100.64.0.0/10",    // Tailscale VPN
    "172.20.0.0/16",    // Kubernetes Service CIDR
}
```

:::warning Предупреждение
Исключения CIDR должны быть максимально точными. Избегайте использования слишком больших диапазонов (например, `0.0.0.0/0`), что равносильно отключению защиты от SSRF.
:::

## Защита от перепривязки DNS

HTTPC использует паттерн «разрешение-проверка-прямое-соединение» для предотвращения атак перепривязки DNS:

1. Разрешение домена в IP-адрес
2. Проверка всех разрешённых IP на принадлежность к приватным адресам
3. Прямое подключение к проверенному IP (а не повторное разрешение домена)

```go
// Сценарий атаки:
// 1. Злоумышленник контролирует DNS для evil.com
// 2. Первое разрешение возвращает публичный IP (проходит проверку)
// 3. При фактическом подключении DNS возвращает 127.0.0.1 (обход проверки)
//
// Защита HTTPC: после проверки используется проверенный IP для прямого соединения, повторное разрешение не выполняется
```

## Проверка SSRF при перенаправлениях

Цели перенаправлений также проходят проверку SSRF:

```go
// Предположим, запрос к public-api.com, сервер возвращает 302 на http://169.254.169.254/
// HTTPC проверит IP цели перенаправления, блокируя доступ к сервису метаданных
```

### Белый список доменов перенаправлений

```go
cfg := httpc.DefaultConfig()
cfg.Security.RedirectWhitelist = []string{
    "api.example.com",
    "auth.example.com",
    "*.cdn.example.com",  // Поддержка шаблонов
}

// Перенаправления на домены, не входящие в белый список, будут заблокированы
```

## Защита метаданных облачных сред

Адреса сервисов метаданных облачных платформ:

| Платформа | Адрес | Описание |
|-----------|-------|----------|
| AWS | `169.254.169.254` | Метаданные инстанса |
| GCP | `metadata.google.internal` | Сервис метаданных |
| Azure | `169.254.169.254` | Метаданные инстанса |
| Alibaba Cloud | `100.100.100.200` | Сервис метаданных |

HTTPC по умолчанию блокирует доступ к метаданным AWS/Azure (`169.254.169.254` находится в блокируемом списке `169.254.0.0/16`). Метаданные GCP (`metadata.google.internal`) блокируются через проверку разрешения DNS.

:::warning Предупреждение
Метаданные Alibaba Cloud (`100.100.100.200`) находятся в диапазоне CGNAT (`100.64.0.0/10`). Для поддержки VPN типа Tailscale/WireGuard этот диапазон **не** включён в список блокировки по умолчанию. Для защиты доступа к метаданным Alibaba Cloud используйте другие политики безопасности (например, правила брандмауэра).
:::

## Полное отключение защиты от SSRF

Используйте только в тестовой среде:

```go
// TestingConfig отключает защиту от SSRF
client, _ := httpc.New(httpc.TestingConfig())

// Или ручная настройка
cfg := httpc.DefaultConfig()
cfg.Security.AllowPrivateIPs = true
```

:::danger Опасность
Никогда не устанавливайте `AllowPrivateIPs = true` в production.
:::

## Лучшие практики

1. Используйте `SecureConfig()` как базовый уровень безопасности
2. Делайте исключения только для необходимых диапазонов CIDR
3. Настройте `RedirectWhitelist` для ограничения целей перенаправлений
4. Регулярно проверяйте конфигурацию `SSRFExemptCIDRs`
5. Используйте промежуточное ПО аудита для логирования всех запросов
